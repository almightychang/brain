# brain:connect — 점진적 지식 연결

## 전제 조건

이 스킬 실행 전 `${CLAUDE_PLUGIN_ROOT}/config.md`를 읽어 VAULT, WORKSPACES 경로를 확인한다.

Obsidian Second Brain에서 문서 간 누락된 연결을 발견하고, **소비자 → 공급자** 원칙에 따라 단방향 위키링크를 추가하는 스킬.

한 번에 모든 연결을 찾지 않는다. 호출할 때마다 소규모 배치로 점진적으로 연결을 늘린다.

## 경로 상수

```
VAULT="{config.md의 VAULT}"
PROJECT="${VAULT}/1. Project"
AREAS="${VAULT}/2. Areas"
RESOURCES="${VAULT}/3. Resources"
ARCHIVE="${VAULT}/4. Archive"
INBOX="${VAULT}/0. Inbox"
```

## 입력

```
/brain:connect                          — 최근 수정 파일 5개를 시드로 사용
/brain:connect <파일명 또는 경로>        — 특정 문서를 시드로 사용
/brain:connect <폴더 경로>              — 폴더 내 문서를 시드로 사용 (최대 5개)
/brain:connect --scope recent           — 최근 7일 내 수정 파일
/brain:connect --scope orphan           — 위키링크가 0개인 고아 문서
```

---

## 연결 방향 원칙: 소비자 → 공급자

위키링크는 **지식을 쓰는 쪽(소비자)에서 지식을 제공하는 쪽(공급자)으로** 건다.

이렇게 하면 Obsidian 백링크 패널에서 "이 지식을 활용하는 문서들"이 자연스럽게 보인다.

### 방향 판단 기준

| 패턴 | 방향 | 판단 근거 |
|------|------|-----------|
| **적용 → 원천** | 구체적 작업 → 이론/개념 | 프로젝트 문서가 개념 문서를 참조 |
| **사례 → 범주** | 개별 사례 → 상위 주제 | 케이스 스터디가 분류 주제를 참조 |
| **파생 → 출처** | 요약/분석 → 원본 | 내가 쓴 글이 원본 자료를 참조 |
| **후행 → 선행** | 이후 작업 → 이전 근거 | 솔루션이 문제 정의를 참조 |
| **좁은 → 넓은** | 세부 주제 → 포괄 주제 | 하위 개념이 상위 개념을 참조 |

### PARA 계층에서의 방향

```
Project → Area        프로젝트가 관리 영역의 맥락을 참조
Project → Resource    프로젝트가 참고 자료를 참조
Area    → Resource    영역이 참고 자료를 참조
Resource → Resource   동등한 개념 간 참조 (더 구체적인 쪽 → 더 일반적인 쪽)
```

**Archive는 연결 대상에서 제외한다.** 이미 끝난 맥락에 새 링크를 거는 것은 노이즈.

### 방향이 모호할 때

두 문서가 동등한 수준이면 (예: Resource ↔ Resource 같은 레벨), **시드 문서 쪽에만** 링크를 추가한다. 양쪽에 추가하지 않는다. 다음 번에 반대쪽 문서가 시드가 되면 그때 역방향이 자연스럽게 발견된다.

## Step 1: 시드 선정

### 인자 없이 호출한 경우

**폴더 분산 + 랜덤화** 전략으로 시드를 선정한다. 같은 폴더에서 여러 시드가 나오면 연결의 다양성이 떨어지므로, PARA 폴더별로 분산한다.

1. 대상 PARA 폴더 목록: `0. Inbox`, `1. Project`, `2. Areas`, `3. Resources`
2. 각 폴더에서 최근 30일 내 수정된 `.md` 파일을 수집한다 (Daily Notes, `.obsidian/`, `5. Plugins/` 제외)
3. 각 폴더 풀에서 **랜덤으로 1~2개**를 뽑되, 같은 하위 폴더(예: `3. Resources/Language - Rust`)에서 2개 이상 뽑지 않는다
4. 총 5개가 될 때까지 채운다. 특정 폴더에 후보가 없으면 다른 폴더에서 추가로 뽑는다

```bash
# 각 PARA 폴더별 후보 수집 (최근 30일, 랜덤 정렬)
for dir in "$INBOX" "$PROJECT" "$AREAS" "$RESOURCES"; do
  find "$dir" -name "*.md" -not -path "*/Daily Notes/*" -not -path "*/.obsidian/*" -not -path "*/5. Plugins/*" -mtime -30 -type f -printf '%T@ %p\n' | sort -R | head -2
done | head -5 | cut -d' ' -f2-
```

**핵심**: 같은 하위 폴더에서 2개 이상 시드가 선정되면, 초과분을 다른 폴더의 후보로 교체한다. 이렇게 하면 매 호출마다 다양한 영역의 문서가 시드로 선정되어 cross-domain 연결을 발견할 수 있다.

### --scope orphan

위키링크(`[[`)가 본문에 하나도 없는 파일을 탐색한다. 이들은 연결이 가장 필요한 문서.

### 파일/폴더 지정

지정된 파일 또는 폴더 내 `.md` 파일을 시드로 사용. 폴더의 경우 최대 5개 (최근 수정 순).

## Step 2: 시드 문서 분석

각 시드 문서를 읽고 다음을 추출한다:

```yaml
seed:
  path: "파일 경로"
  para_level: "Project | Area | Resource"
  title: "문서 제목"
  keywords: ["핵심 키워드 3-5개"]
  concepts: ["문서가 다루는 주요 개념 2-3개"]
  existing_links: ["이미 존재하는 위키링크 목록"]
  role: "consumer | provider | peer"
```

### role 판단

- **consumer**: 다른 지식을 적용/활용하는 문서 (Project 문서, 분석 문서, 의사결정 문서)
- **provider**: 지식을 제공하는 문서 (개념 설명, 레퍼런스, 논문 요약)
- **peer**: 동등한 수준의 주제 문서 (같은 Resource 폴더 내 형제 문서)

시드가 여러 개면 Agent를 사용하여 **병렬로** 분석한다.

## Step 3: 연결 후보 탐색

시드 문서별로 연결 후보를 탐색한다. Agent를 사용하여 병렬 처리.

### 3-1. 탐색 범위

시드의 `role`과 `para_level`에 따라 탐색 범위를 결정한다:

| 시드 role | 탐색 대상 |
|-----------|-----------|
| consumer (Project) | Areas, Resources 전체 |
| consumer (Area) | Resources 전체 |
| provider (Resource) | 같은 Resources 폴더 + 인접 주제 폴더 |
| peer | 같은 상위 폴더 + 관련 주제 폴더 |

### 3-2. 탐색 방법

순서대로 수행. 이전 단계에서 충분한 후보가 나오면 (5개 이상) 다음 단계를 생략한다.

1. **폴더명 매칭** — 키워드와 관련된 폴더 식별
2. **파일명 매칭** — Glob으로 키워드가 파일명에 포함된 파일 탐색
3. **본문 키워드 검색** — Grep으로 핵심 키워드 검색
4. **개념 수준 검색** — concepts에서 파생된 유의어/관련어로 확장 검색

### 3-3. 필터링

- `existing_links`에 이미 포함된 문서는 **제외**
- Archive 폴더 문서는 **제외**
- Daily Notes는 **제외**
- 시드 문서 자신은 **제외**

### 3-4. 관련도 평가

각 후보 문서의 제목과 첫 10줄(또는 frontmatter + 첫 단락)을 읽고 평가한다:

```yaml
candidate:
  path: "후보 문서 경로"
  title: "문서 제목"
  relevance: "strong | moderate"
  direction: "seed → candidate | candidate → seed"
  pattern: "적용→원천 | 사례→범주 | 파생→출처 | 후행→선행 | 좁은→넓은 | peer"
  reason: "연결 이유 한 줄"
```

- `strong`: 명확한 개념적 연결이 있는 경우
- `moderate`: 간접적이지만 유용한 연결
- weak는 수집하지 않는다 — **노이즈보다 precision을 우선한다**

### 3-5. 방향 결정

`direction` 필드는 **어느 문서에 링크를 추가할지**를 결정한다:

- `seed → candidate`: 시드 문서에 `[[candidate]]` 링크를 추가
- `candidate → seed`: 후보 문서에 `[[seed]]` 링크를 추가

방향은 Step 2의 role과 연결 방향 원칙에 따라 결정:

1. consumer 쪽에 링크를 추가한다
2. 둘 다 provider면, 더 구체적인 쪽에 추가한다
3. 판단이 모호하면 시드 쪽에 추가한다 (현재 작업 맥락이므로)

## Step 4: 연결 리포트 출력

시드 문서별로 발견된 연결을 보고한다:

```markdown
## 🔗 Connect: 연결 발견 리포트

**시드**: N개 문서 분석 | **발견**: M개 연결 후보

---

### 📄 {시드 문서 제목}
> `{시드 경로}` | role: {consumer/provider/peer} | 기존 링크: {N}개

| # | 링크 삽입 (📝 = 편집 대상) | 패턴 | 강도 | 이유 |
|---|---------------------------|------|------|------|
| 1 | 📝 **시드 제목** → [[대상 제목]] | 적용→원천 | strong | 이유... |
| 2 | 📝 **시드 제목** → [[대상 제목]] | 사례→범주 | moderate | 이유... |
| 3 | 📝 **대상 제목** → [[시드 제목]] | 좁은→넓은 | strong | 이유... |

`📝`가 붙은 문서에 `[[타겟]]` 위키링크가 추가된다. 화살표(→)는 소비자→공급자 방향.

---

### 📄 {시드 문서 2 제목}
...

---

### 📊 요약
- 분석한 시드: N개
- 발견한 연결: M개 (strong: X, moderate: Y)
- 시드에 추가: A개
- 대상에 추가: B개
```

리포트 출력 후 사용자에게 안내:

```
다음 중 선택해주세요:
- "전체" — 모든 연결 추가
- "1-1, 1-3, 2-2" — 시드번호-연결번호로 선택
- "strong만" — strong 연결만 추가
- "취소" — 아무것도 하지 않음
```

## Step 5: 링크 삽입

사용자가 승인한 연결에 대해 위키링크를 삽입한다.

### 삽입 위치

문서 하단의 `## Related` 섹션에 추가한다.

**`## Related` 섹션이 이미 있는 경우:**
- 기존 항목 아래에 새 링크를 추가한다
- 이미 같은 링크가 있으면 스킵한다

**`## Related` 섹션이 없는 경우:**
- 문서 맨 끝에 빈 줄 + `## Related` + 링크를 추가한다

### 링크 형식

```markdown
## Related

- [[대상 문서 제목]] — 연결 이유 (한 줄)
```

- 위키링크는 **파일명만** 사용한다 (전체 경로 X). Obsidian이 자동으로 resolve.
- 별칭이 필요한 경우 `[[파일명|표시명]]` 사용
- 연결 이유는 `—` (em dash) 뒤에 간결하게

### 삽입 예시

**Before:**
```markdown
# GPU 추론 서비스 아키텍처

본문 내용...
```

**After:**
```markdown
# GPU 추론 서비스 아키텍처

본문 내용...

## Related

- [[Kubernetes 리소스 관리]] — 추론 파드 스케줄링 전략의 기반
- [[모델 서빙 비용 최적화]] — 비용 관점에서의 아키텍처 제약
```

### 삽입 완료 보고

```
✅ 연결 완료
- 📄 {문서1}: +2개 링크 추가
- 📄 {문서2}: +1개 링크 추가 (1개 중복 스킵)
- 📄 {문서3}: +1개 링크 추가
총 4개 연결 추가 (1개 스킵)
```

## 원칙

- **소비자 → 공급자**: 지식을 쓰는 쪽에서 제공하는 쪽으로 링크한다. 이것이 유일한 방향 원칙.
- **precision > recall**: 약한 연결은 버린다. 노이즈 링크는 없는 것보다 나쁘다.
- **점진적**: 한 번에 5개 시드, 시드당 최대 5개 연결. 매일 조금씩 연결을 늘리는 게 목표.
- **비파괴적**: 기존 본문을 수정하지 않는다. `## Related` 섹션만 추가/확장한다.
- **사용자 결정 우선**: 링크 삽입은 반드시 사용자 확인 후 실행한다.
- **Archive 제외**: 완료된 프로젝트에 새 링크를 거는 것은 노이즈.
- **중복 방지**: 이미 존재하는 위키링크(본문 어디에든)는 다시 추가하지 않는다.
